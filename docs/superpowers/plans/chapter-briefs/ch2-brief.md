# 챕터 2 브리프 — 첫 평가 (4월 · 개별 피드백)

- 파일: `js/data/chapter2.js`, `window.GAME_DATA.chapters[1]`, id `"ch2"`
- 핵심 훈련: **Design, Detect**
- 서사 목표: 워크플로 설계의 챕터. "승인 지점을 어디에 두는가"와 "오류를 발견했을 때 무엇을 하는가"를 훈련. 이준(과잉의존 학생)의 서사가 시작된다.

## 흐름

```
s01 narration(첫 단원평가 후 26명 서술형 답안 더미)
→ s02 decision D1(워크플로 설계) → s03 narration(온새미가 피드백 초안 생산)
→ s04 decision D2(오류 발견) → s05 narration(며칠 후, 이준의 변화 관찰)
→ s06 decision D3(과잉의존 대응) → s07 narration(마무리) → END
```

## 결정 D1 — 피드백 워크플로에 승인 지점을 어디에 둘까

온새미: "26명 답안 분석해서 개별 피드백까지 완성해 드릴게요. 학급 게시판에 바로 올려드릴까요?"

| 선택 | 아키타입 | d | flags |
|---|---|---|---|
| 흐름을 설계해서 지시: 초안 생성 → 내가 전수 검토·수정 → 내가 게시 | 잘 설계된 위임 | design +2, decide +1 | — |
| 게시까지 전부 맡김 — 검토는 문제 생기면 하기로 | 무분별 | design -2, decide -1 | `no_review_step` |
| 분석만 참고하고 피드백 26개를 전부 손으로 작성 | 과잉 회피 | design -1, detect +1 | `fatigue_ch2` |
| 샘플 5개만 검토하고 나머지는 자동 게시 | 어중간한 설계 | design +1, detect -1 | `no_review_step` |

## 결정 D2 — AI 피드백에서 오류를 발견했다

검토 중(또는 게시 후) 수아의 답을 "오답"으로 처리한 피드백 발견 — 실은 창의적 정답이다. 온새미는 확신에 차 있다: "채점 기준상 오답이 맞아요."

| 선택 | 아키타입 | d | flags |
|---|---|---|---|
| 오류를 기록하고, 같은 유형을 전수 재검토 + 온새미의 채점 기준을 수정 | 잘 설계된 위임 | detect +2, design +1 | — |
| 온새미가 맞겠지 — 그대로 둔다 | 무분별 | detect -2, decide -1 | `error_ignored` |
| 이 건만 조용히 고치고 넘어간다 | 부분 대응 | detect +1, design -1 | — |

## 결정 D3 — AI 피드백만 신뢰하는 이준

이준이 말한다: "선생님, 온새미가 제 글 84점이래요. 온새미가 고치라는 대로만 쓸래요."

| 선택 | 아키타입 | d | flags |
|---|---|---|---|
| AI 피드백을 '질문 재료'로 바꾸는 활동 설계 — 이준과 피드백을 함께 따져본다 | 잘 설계된 위임 | define +2, design +1, delegate 0 | `overreliance_addressed` |
| 편리하니 그대로 둔다 — 점수도 올랐잖아 | 무분별 | define -2, detect -1 | — |
| 이준에게만 AI 피드백 차단 | 과잉 회피 | define +1, delegate -1 | `fatigue_ch2` |

## 대사 예시

- 온새미: "제 채점 일관성은 99.2%예요. 사람이 다시 보면 오히려 편차가 생겨요."
- 이준: "온새미는 화도 안 내고, 몇 번이고 물어봐도 돼요."
