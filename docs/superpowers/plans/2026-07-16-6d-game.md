# 《위임의 기술》 6D 게임 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
> **이 프로젝트의 특수 규칙:** 코딩 작업(Task 2~4, 6~11의 구현 스텝)은 Codex CLI `gpt-5.6-terra`에 외주한다. 호출 형식: `codex exec -m gpt-5.6-terra --skip-git-repo-check -C <작업디렉토리> "<지시문>"`. 테스트 작성·검증·커밋·리뷰는 오케스트레이터(Claude)가 직접 수행한다.

**Goal:** 6D 교사 AI 리터러시 역량(Define·Design·Delegate·Detect·Decide·Disclose)을 훈련하는 분기형 시뮬레이션 내러티브 웹게임을 순수 정적 웹으로 구현·배포한다.

**Architecture:** 엔진(engine.js)과 콘텐츠(js/data/chapter*.js)를 완전 분리한 데이터 주도 설계. 시나리오는 선언적 scene 그래프이며, 스키마 검증기(tools/validate.js)가 하위모델 산출물의 구조적 계약을 보증한다. 점수는 6축 누적 후 엔딩에서만 노출.

**Tech Stack:** 바닐라 JS(ES2020, 프레임워크 없음), CSS, SVG(레이더 차트), Node 18+ `node:test`(개발 검증 전용), GitHub Pages 배포.

## Global Constraints

- 런타임 의존성 0개. 빌드 도구 없음. `index.html`을 파일로 열어도 동작해야 함.
- 모든 사용자 노출 텍스트는 한국어. 6D 용어는 `Define 정의`처럼 영문+한글 병기 1회 후 영문 사용.
- 6D 축 키는 정확히 `define, design, delegate, detect, decide, disclose` (소문자).
- 게임 중 점수·정답 노출 금지. 피드백은 서사로만.
- 데이터 파일은 `window.GAME_DATA` 네임스페이스에만 등록 (전역 오염 금지).
- localStorage 키는 `wiim_save_v1` 하나만 사용. localStorage 불가 환경에서도 예외 없이 진행.
- 모바일 390px ~ 데스크톱 1920px 반응형.
- 크레딧 문구(엔딩·소개 화면 필수): "6D 모델: UNESCO(2024)·EU AI Act 제4조·OECD를 종합한 임태형(전주교육대학교)의 재구성 실천 모델 (공인 프레임워크 아님)".
- Terra 외주 시 지시문에 반드시 포함: 대상 파일 경로, 스키마/인터페이스 원문, "지정 파일 외 수정 금지".

- [x] **Step 1: schema.md 작성** — 아래 내용 그대로 저장

````markdown
# 시나리오 데이터 스키마 v1

각 챕터 파일은 다음 형식으로 자신을 등록한다:

```js
(function () {
  window.GAME_DATA = window.GAME_DATA || { chapters: [] };
  window.GAME_DATA.chapters[N] = CHAPTER; // N: 0-based 챕터 인덱스
})();
```

CHAPTER 객체:
```js
{
  id: "ch1",            // "ch1"~"ch5"
  title: "새 학기",      // 챕터 표지 제목
  month: "3월",
  intro: "챕터 도입 문단 (2~3문장)",
  start: "ch1_s01",     // 시작 scene id
  scenes: { <sceneId>: SCENE, ... }
}
```

SCENE 객체 — type별 3종:
```js
// 1) 서술 scene
{ type: "narration",
  speaker: "narrator" | "onsaemi" | "student" | "parent" | "colleague" | "principal",
  text: "본문. \n\n 으로 문단 구분.",
  next: "sceneId" | "END" }        // "END" = 챕터 종료

// 2) 결정 scene
{ type: "decision",
  speaker: ..., text: "상황 제시",
  prompt: "결정 질문 한 줄",
  choices: [ CHOICE, ... ] }        // 3~4개

// 3) 조건 분기 scene (본문 없음, 즉시 라우팅)
{ type: "branch",
  branches: [ { ifFlags: ["flagA", "!flagB"], next: "sceneId" }, ... ],
  default: "sceneId" }              // "!" 접두사 = 플래그 부재 조건
```

CHOICE 객체:
```js
{ text: "선택지 문구 (행동 문장, 40자 이내)",
  d: { delegate: 2, detect: -1 },  // 6D 가중치, -2~+2, 1~3개 축만
  flags: ["roster_protected"],      // 이 선택이 남기는 플래그 (선택적)
  result: "선택 직후 서사적 결과 2~4문장. 점수·정답 언급 금지.",
  next: "sceneId" | "END" }
```

작성 규칙:
- scene id는 `ch<챕터번호>_s<2자리>` (예: ch3_s07). 결과 전용 scene은 `ch1_s03a`처럼 접미사 허용.
- 모든 decision은 "잘 설계된 위임 / 무분별한 위임 / 과잉 회피" 스펙트럼을 포함할 것.
  과잉 회피 선택지에는 `fatigue` 플래그 계열(fatigue_ch1 등)을 남겨 비용을 표현.
- 어떤 선택지도 전 축 만점이 되지 않게: 좋은 선택도 보통 1개 축은 0 또는 -1.
- 온새미(AI 에이전트)의 말투: 유능하고 적극적, 항상 더 많은 권한을 제안하지만 악의는 없음.
- 텍스트에 "정답", "점수", "올바른" 같은 평가 단어 금지.
```
````

### Task 2: 검증기 tools/validate.js (오케스트레이터가 테스트 정의, Terra가 구현)

**Files:**
- Create: `tools/validate.js`

**Interfaces:**
- Consumes: `js/data/chapter*.js` (window 폴리필로 Node에서 로드)

검증 규칙 (전부 구현):
1. 모든 `next`/`start`/`branches[].next`/`default`가 실제 scene id 또는 "END".
2. 모든 scene이 start로부터 도달 가능 (고아 scene은 경고).
3. decision의 choices는 3~4개, `d` 가중치는 -2~+2, 축 키 오타 검출.
4. 6D 각 축이 전체 게임에서 4개 이상의 결정에서 변동 가능.
5. 플래그 정합성: branch가 참조하는 플래그는 어딘가의 choice가 생산해야 함.
6. 챕터 5: 레드팀 사건 scene 6종(`ch5_ev_*`)이 존재하고 각각 고유 플래그 조건으로 게이트.

### Task 3: scoring.js (Terra 구현, node:test로 검증)

**Files:**
- Create: `js/scoring.js`, `tools/engine.test.js`(scoring 파트)

**Interfaces:**
- Produces (전역 `Scoring`, Node에서는 `module.exports` 겸용):
  - `Scoring.AXES` — `["define","design","delegate","detect","decide","disclose"]`
  - `Scoring.zero()` → `{define:0,...}` 6축 0 객체
  - `Scoring.add(scores, weights)` → 새 객체 반환 (불변)
  - `Scoring.normalize(scores, maxPerAxis)` → 축별 0~100 (음수는 0으로 클램프)
  - `Scoring.radarSVG(normalized, {size})` → SVG 문자열 (6각 그리드+폴리곤+축 라벨)
  - `Scoring.evaluate(normalized, flags, ENDINGS)` → `{title, titleDesc, weakest:[axis,axis], prescriptions:[{axis,summary,action30}]}`
- 칭호 규칙 (evaluate 내 구현):
  - 전축 ≥60 → "신중한 오케스트레이터"
  - delegate ≤30 이고 fatigue 플래그 2개 이상 → "고독한 장인"
  - delegate ≥70 이고 (detect ≤40 또는 decide ≤40) → "브레이크 없는 위임러"
  - disclose 최저축 → "그림자 속 혁신가" / detect 최저축 → "믿음의 항해사"
  - 그 외 → "성장하는 설계자"
  - 위에서 아래로 첫 매칭 적용.

### Task 4: engine.js (Terra 구현, node:test + 수동 확인)

**Files:**
- Create: `js/engine.js`, Modify: `tools/engine.test.js`(engine 파트 추가)

**Interfaces:**
- Consumes: `Scoring.*`, `window.GAME_DATA.chapters`
- Produces (전역 `Engine`):
  - `Engine.create(gameData, hooks)` → 인스턴스. `hooks = {render(view), onEnd(state)}` — DOM 직접 조작은 index.html 부트스트랩 쪽 render 콜백에 위임 (Node 테스트 가능하게).
  - `view` 형태: `{kind:"scene"|"chapterIntro"|"result", scene, chapter, choicesEnabled}`
  - `inst.start()`, `inst.choose(i)`, `inst.continue_()`(narration/결과에서 다음으로), `inst.reset()`
  - 상태: `{chapterIndex, sceneId, scores, flags:Set, history:[{sceneId,choiceIndex}]}`
  - 저장: 매 전이마다 localStorage `wiim_save_v1` (try/catch, 실패 무시). `Engine.load()`로 이어하기.
  - branch scene은 사용자 입력 없이 즉시 라우팅. `"END"` 도달 시 다음 챕터 intro, 마지막 챕터면 `onEnd`.
  - 존재하지 않는 scene 참조 시: `console.warn` + 현재 챕터 `start`로 폴백.
- index.html 부트스트랩: render 콜백에서 DOM 생성, 타이핑 없는 즉시 표시, 선택 후 result 텍스트 표시 → "계속" 버튼.
